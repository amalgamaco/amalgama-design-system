Auditoría de patrón: Icon Buttons

Embassy DS · Verificación contra Button, Color Roles, States, y Governance · Jun 2026

Conclusión en una línea: .search-icon-btn está referenciado como sub-patrón del componente Search, pero NO está especificado como un Icon Button variant en el sistema de botones. Tiene documentación parcial (existe en docs/search.html) pero le falta la especificación de estados, tokens, y el racional de diseño que justifique la forma circular y la separación de .icon-btn. No inventa tokens, no viola Color Roles — pero sí introduce una variante de botón fuera del árbol de Button, sin justificación documentada, y sin mención en GOVERNANCE. Es un gap de documentación y un riesgo de fragmentación.

El problema central: dos clases para icon buttons

Actualmente existen dos definiciones de icon button que coexisten sin jerarquía documentada entre ellas:

Button family .icon-btn

  • Archivo: button.css
  • Tamaño: 36×36px (MD default)
  • Forma: --radius-md = 8px (rectangular)
  • Fondo: --color-surface-container
  • Borde: 1px solid var(--color-outline)
  • Ícono: --color-on-surface-variant
  • Selected: no definido
  • Documentación: ✓ sección completa en index.html
  • Pertenece a: jerarquía de Button (5ta variante)

Search family .search-icon-btn

  • Archivo: search.css
  • Tamaño: 40×40px
  • Forma: border-radius: 50% (circular)
  • Fondo: --color-surface-container-high
  • Borde: 1px solid var(--border) (más sutil)
  • Ícono: --color-on-surface-variant
  • Selected: --color-secondary-container ✓
  • Documentación: ⚠ fila de tabla en docs/search.html
  • Pertenece a: componente Search (sub-patrón)

La diferencia más importante es la forma y el tamaño. .icon-btn usa el radius scale de Button (§4.3 de GOVERNANCE). .search-icon-btn usa border-radius: 50%, que no está en el radius scale ni en la size table de botones — es la forma de avatares y la píldora de chips, aplicada a un elemento cuadrado.

¿La forma circular viola el sistema de formas?

GOVERNANCE §4.2: "A button with --radius-full is visually indistinguishable from a chip. The pill shape communicates 'selectable tag' — using it on a button breaks hierarchy and misleads users about the element's role. Buttons never use --radius-full."

El §4.2 prohíbe --radius-full en botones para evitar confusión con chips. Pero .search-icon-btn no usa --radius-full — usa border-radius: 50%. En un elemento cuadrado (40×40), esto produce un círculo perfecto, que es visualmente distinto de la píldora de un chip (que requiere que el elemento sea más ancho que alto). Técnicamente no viola la letra de §4.2.

Sin embargo, border-radius: 50% no está en el radius scale de GOVERNANCE §4.1 (sm/md/lg/xl/full). GOVERNANCE §4.4 dice: "A component must use a radius token, not a computed or hardcoded intermediate value. If no token fits, this is a DS gap." El círculo es un valor hardcoded que no tiene token propio — esto sí es técnicamente una violación de §4.4.

La excepción razonable: avatares también usan border-radius: 50%. El DS acepta implícitamente este valor para elementos circulares. Pero esa excepción no está documentada en GOVERNANCE.

Hallazgos

✗

No está documentado como Icon Button variant

.search-icon-btn no aparece en el Button section de index.html. Cualquier consumidor que busque "cómo hacer un icon button circular con selected state" en la documentación canónica no lo va a encontrar. Eso es un gap de documentación que produce exactamente el comportamiento que describís: alguien inventa un patrón porque no encontró el existente.

✗

GOVERNANCE §12.4 ya registra icon button fragmentation — .search-icon-btn no fue capturado

La tabla de "Known Inconsistencies" lista 5 implementaciones fragmentadas de icon buttons. .search-icon-btn es una sexta que no fue capturada en ese audit. Esto confirma que el problema que señalás es real y ya existía antes de este componente.

✗

border-radius: 50% no es un radius token — viola GOVERNANCE §4.4

El radius scale del DS es sm/md/lg/xl/full. Ninguno de esos tokens aplicado a un elemento cuadrado produce un círculo. El círculo usa un valor hardcoded sin equivalente en el sistema. Esto no está exceptuado en GOVERNANCE (a diferencia de los avatares, que comparten la misma situación pero tampoco tienen excepción documentada).

!

No hay state demos ni token table en Search docs

La documentación de .search-icon-btn en docs/search.html es una fila de tabla con una descripción de una línea. No tiene estado demos (default, hover, focus, pressed, selected, disabled), no tiene token table (qué token usa en cada estado), y no tiene racional de diseño. Esto viola el estándar de documentación de GOVERNANCE §10.

!

No hay regla que defina cuándo usar .icon-btn vs .search-icon-btn

Un consumidor que necesita un icon button circular con selected state no sabe si debe usar .icon-btn + algún modifier, .search-icon-btn fuera del contexto de search, o algo distinto. Esta ambigüedad es la fuente del patrón "inventado" que describís.

✓

Los tokens usados son correctos — no hay violaciones de Color Roles

El componente consume roles semánticos correctos para cada estado. El visual treatment (neutral fill + border + stroke icon + secondary-container selected) es coherente con los Color Roles definidos en variables.css y con el patrón de selected state documentado en GOVERNANCE §5.5.

✓

El patrón base (neutral fill + border + icon) sí está establecido por .icon-btn

La combinación visual "fondo de superficie + borde outlined + ícono de trazo" no es nueva. Es exactamente lo que .icon-btn hace. .search-icon-btn no inventa un tratamiento visual — lo adapta con forma circular y selected state para el contexto de búsqueda. El problema no es el visual, es que esta adaptación no está especificada formalmente.

¿Qué hacer? Tres opciones

✓ Recomendado
Opción A: Formalizar como sub-componente de Search con documentación completa

Mantener .search-icon-btn en search.css, pero documentarlo correctamente:

  • Agregar state demos en docs/search.html
  • Agregar token table por estado
  • Documentar el racional de la forma circular (acompaña el pill shape del search-bar)
  • Agregar border-radius: 50% como excepción explícita en GOVERNANCE §4.4 (igual que avatares)
  • Registrar en GOVERNANCE §12.4 como issue conocido: "search-icon-btn no está cross-referenciado desde Button"
  • Agregar cross-referencia desde Button section de index.html
~ Alternativa
Opción B: Consolidar en el sistema de botones como .icon-btn modifier

Mover el patrón al árbol de Button como una variante formal:

  • Agregar .icon-btn.icon-btn-circular o .icon-btn--search en button.css
  • Mover los estilos de search.css a button.css
  • Documentar en el Button section de index.html
  • Requiere decidir si "circular" es una variante de Button o una propiedad de contexto
  • Más difícil de justificar: el círculo es semánticamente correcto junto a un search-bar, no como icon button genérico
✗ No recomendado
Opción C: Reemplazar con .icon-btn existente (rectangular)

Eliminar .search-icon-btn y usar .icon-btn.btn-lg (44px rectangular):

  • Simplifica el sistema
  • Pero rompe la coherencia visual: un search-bar circular con botones rectangulares crea tensión de forma
  • Requiere decisión de diseño explícita
  • El selected state de .icon-btn tampoco está definido — necesitaría la misma work

Acciones concretas

1

Registrar en GOVERNANCE §12.4 — alta prioridad

Agregar .search-icon-btn a la tabla de "Icon button fragmentation" en §12.4 con la nota: "Sub-variante circular del icon button, definida en search.css. Requiere cross-referencia desde Button section y decisión sobre si consolida en button.css o se formaliza como sub-componente de Search."

2

Agregar state demos a docs/search.html — documentación mínima

Sin state demos formales, cualquier implementación futura puede divergir. Se necesita al menos: default, hover, focus, pressed, selected, disabled — con los tokens correspondientes.

3

Documentar el racional de la forma circular en GOVERNANCE

GOVERNANCE §4.4 prohíbe valores de radius fuera del scale. Hay que agregar una excepción documentada para border-radius: 50% en elementos cuadrados (avatar, search-icon-btn) con la justificación de diseño.

4

Cross-referenciar desde la sección Button de index.html

El Button section debe mencionar que existe un icon button circular para contextos de búsqueda (.search-icon-btn) y apuntar a la sección Search. Sin esa referencia, cualquier consumidor que busque "icon button con selected state" no lo encuentra.

5

Decisión de diseño sobre .icon-btn vs .search-icon-btn (cuándo usar cada uno)

Falta una regla explícita: "Usá .icon-btn cuando... Usá .search-icon-btn cuando...". Propuesta: .icon-btn para acciones inline en tablas, listas, y toolbars rectangulares. .search-icon-btn exclusivamente en .search-row — patrón circular que acompaña la forma pill del search-bar.
