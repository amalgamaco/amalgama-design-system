§00 Visión general

Este informe evalúa el estado actual de Embassy — el sistema de diseño de Amalgama — como base para construir productos digitales escalables, white-label y consumibles por herramientas de IA. El análisis cubre la documentación existente, la consistencia entre componentes y tokens, y los gaps que bloquearían una adopción confiable.

La evaluación se realizó revisando todos los archivos .md, los 22 archivos CSS de componentes, las 17 implementaciones React, el sitio de documentación, el SKILL.md del skill de IA, y comparando las reglas del sistema contra las implementaciones reales.

§02 Readiness: construir un producto nuevo desde cero
LISTO CON RESERVAS

Embassy es suficientemente maduro para construir un producto nuevo. Los tokens, la mayoría de los componentes, y la documentación de consumo están en buen estado. Las reservas son inconsistencias que se manifiestan como deuda técnica visible, no como bloqueantes de funcionalidad.

  • ✓ Token system completo y bien estructurado. Dark mode automático funcionando.
  • ✓ 22 componentes CSS consumibles, 17 wrappers React.
  • ✓ CLAUDE.md es suficiente para onboardear un dev sin ambigüedad.
  • ✓ App shell (sidebar + topbar) implementado en layout.css.
  • ! Roadmap de componentes extenso: checkbox, radio, switch, menú, tooltip, date picker — sin código. Productos que los necesiten deben componer desde existentes.
  • ! segmented-button.css usa --ds-* tokens no definidos. El componente no renderiza correctamente hasta que se corrija.
  • ! Focus ring ausente en search-bar y search-field (WCAG fail). Usar en productos con accesibilidad requerida implica fix previo.
  • ! Inconsistencias de hover (--bg vs --color-surface-variant) y espaciado off-scale son deuda visible pero no bloquean el desarrollo.
§03 Readiness: implementaciones white-label
LISTO CON CONDICIONES

El token system de Embassy está arquitecturalmente preparado para white-labeling — la separación en primitivos → roles semánticos → componentes es la base correcta. Sin embargo, antes de este audit no existía documentación del proceso, y hay CSS bugs que afectan la confiabilidad del theming.

  • ✓ Arquitectura de tokens (primitivos → roles semánticos) es correcta para white-label.
  • ✓ Dark mode vía data-theme="dark" funciona sin overrides por componente — el theming cascadea automáticamente.
  • ✓ WHITE-LABEL.md creado en este audit — proceso completo documentado con template y worked example.
  • ✓ GOVERNANCE.md §15 establece las reglas de theming y lo que es / no es overrideable.
  • ✗ Bloqueante: avatar gradients en vacancy-card.css y person-card.css usan primitivos directamente (--error-600, --secondary-900) — bypassean el sistema semántico. Al cambiar la paleta de un cliente, los avatares NO se actualizarán. Requiere CSS fix.
  • ✗ Bloqueante: segmented-button.css usa --ds-* tokens no definidos. El componente se rompe en cualquier producto.
  • ! El focus ring (--color-focus) es intencionalmente fijo (Embassy blue). WHITE-LABEL.md documenta cómo manejarlo si el cliente requiere un focus color de marca.

§07 Inconsistencias críticas — bloquean theming o accesibilidad
CRITICAL

Estas inconsistencias deben corregirse antes de usar Embassy como base para cualquier white-label o antes de declarar accesibilidad WCAG AA.

✗ CRÍTICOsegmented-button.css usa 6 tokens no definidos (--ds-*)
Los tokens --ds-card, --ds-card-border, --ds-text-1, --ds-text-2, --ds-accent, --ds-accent-bg no existen en variables.css. El componente renderiza sin colores — fondo blanco sobre blanco en light mode, sin contraste en dark.
Fix: En segmented-button.css, reemplazar: --ds-card → var(--card-bg), --ds-card-border → var(--border), --ds-text-1 → var(--text-primary), --ds-text-2 → var(--text-secondary), --ds-accent → var(--color-secondary), --ds-accent-bg → var(--color-secondary-container).
✗ CRÍTICOAvatar gradients usan primitivos — bypassean theming
vacancy-card.css y person-card.css usan --error-600, --secondary-900, --success-500, --tertiary-900 directamente en gradientes. Al aplicar una paleta de marca cliente, los avatares no cambiarán de color. Además, --color-on-primary se usa como text color para TODOS los avatares independientemente de su background — esto es semánticamente incorrecto y puede resultar en texto ilegible si el background no es --color-primary.
Fix: Reemplazar gradientes con fills planos usando pares semánticos: variant default → --color-error-container + --color-on-error-container; .blue → --color-secondary-container + --color-on-secondary-container; .green → --color-success-container + --color-on-success-container; .purple → --color-tertiary-container + --color-on-tertiary-container.
✗ CRÍTICO · A11Ysearch-bar y search-field no tienen focus ring — WCAG fail
.search-bar:focus-within usa solo un cambio de background (no ring). .search-field:focus-within hace la border más clara — el foco hace el elemento menos visible, no más. Ambos son WCAG 2.4.7 fails.
Fix search-bar: Agregar outline: 2px solid var(--color-focus); box-shadow: 0 0 0 4px var(--color-focus-ring); en :focus-within.
Fix search-field: Reemplazar el border-color change con el ring estándar.
