Color

El sistema de color de Embassy está construido sobre paletas primitivas definidas en Figma y organizadas en roles semánticos. Tres niveles de documentación: las paletas base, el sistema que las conecta, y los roles que las aplican a la UI.

Baseline
Los valores primitivos: Primary, Secondary, Tertiary, Information, Neutral y señales semánticas. Todas las escalas de 10 pasos desde Figma.
Color Roles
El propósito de cada color en la interfaz: fondos, texto, contenedores, bordes y estados. El mapa de uso que guía a diseñadores y desarrolladores.

Principios de color

✓ Do
Usá tokens semánticos, no primitivos
En componentes, siempre usá --color-primary en lugar de --primary-900. Los tokens semánticos conectan la decisión de diseño con la implementación.
✗ Don't
No hardcodees valores hex en componentes
Escribir #01164D directamente en CSS rompe la cadena de tokens y hace imposible el theming y el dark mode.
✓ Do
Respetá el par color / on-color
Si usás --color-primary como fondo, el texto encima debe usar --color-on-primary. Esto garantiza contraste accesible en todos los modos.
✗ Don't
No mezcles roles de diferentes familias
No pongas texto --color-on-secondary sobre un fondo --color-primary. Los pares on/container están diseñados para usarse juntos.

Color Roles

Un color role es un nombre que describe para qué sirve un color en la interfaz —una acción principal, un fondo, un texto, un borde— en lugar de describir qué color es. Esta página explica cada rol del sistema de Embassy: qué representa, cuándo usarlo, cuándo no, y cómo se comporta en Light y Dark Mode.

General Concepts

Cuando un diseñador o un developer tiene que elegir un color, la pregunta correcta no es "¿qué tono de azul uso?", sino "¿cuál es el rol de este elemento?". Los color roles responden esa pregunta: encapsulan una decisión de diseño y la nombran, de modo que el mismo rol se aplique igual en toda la suite de productos. Esto crea consistencia, soporta accesibilidad (cada rol viene con su par de contraste) y hace que el dark mode funcione sin tocar los componentes.

Entendiendo la terminología de los color roles

Estos son algunos términos que vas a encontrar a lo largo del sistema de color. Conocerlos ayuda a leer cualquier rol antes de entrar en cada categoría.

Surface
Rol usado para fondos y áreas grandes de baja jerarquía de la interfaz. Las superficies crean la base sobre la que se muestran el contenido y los componentes.
Primary · Secondary · Tertiary
Accent color roles que establecen jerarquía y énfasis dentro de la UI. Ayudan a identificar acciones importantes, elementos interactivos y zonas de foco.
Container
Roles usados como color de relleno de elementos en primer plano —botones, chips, badges y estados seleccionados—. No deben usarse directamente para texto ni íconos.
On
Los roles que empiezan con "On" definen el color del contenido que va sobre otro rol. Se usan típicamente para texto, íconos e indicadores. Por ejemplo, On Primary es el color del texto y los íconos que se muestran sobre un fondo Primary.
Variant
Los roles que terminan en "Variant" ofrecen una alternativa de menor énfasis a su rol base. Son útiles cuando hace falta jerarquía visual adicional sin introducir un color nuevo. Por ejemplo, Outline Variant es una versión más sutil del rol Outline estándar.

Jerarquía de énfasis · Primary › Secondary › Tertiary

Los tres accent roles no son tres colores intercambiables: forman una escala de énfasis descendente. Lo decisivo es que la jerarquía la define el uso —dónde y cuánto aparece cada rol—, no solo el color. Primary (navy) y Secondary (agile blue) pertenecen a la misma familia azul por identidad de marca; lo que los mantiene distintos es para qué se usa cada uno, no qué tan vibrante es.

Regla principal: nunca como pares. Primary y Secondary no deben competir en el mismo nivel. En una pantalla hay una sola acción Primary dominante; Secondary acompaña como soporte interactivo (links, chips, estados), nunca como un segundo CTA relleno del mismo peso. Cuando dos elementos azules pelean por la atención, el problema casi siempre es de uso, no de color.

Por qué la distinción se apoya en el uso. Primary y Secondary comparten el hue azul de la marca, así que la separación no viene de "apagar" Secondary, sino de la jerarquía de aplicación: Primary domina por presencia (relleno sólido, posición y una sola instancia por pantalla) y Secondary apoya. Tertiary aporta el contraste de hue (púrpura) para momentos puntuales. Respetar estas reglas es lo que hace que la jerarquía se lea con claridad, sin tener que sacrificar la vivacidad de Secondary ni de Tertiary.

Primary

Accent color role · jerarquía alta. Primary es el color institucional navy de Embassy y el de mayor énfasis del sistema. Se reserva para los elementos más importantes de una pantalla —la acción principal, el FAB, la barra de navegación activa y todo lo que el usuario debe notar primero—. Como es el color más fuerte, conviene usarlo con moderación: si todo es primary, nada lo es.

Ejemplos. El botón primario usa Primary + On Primary; el FAB, los estados seleccionados y los componentes destacados usan Primary Container + On Primary Container (un fill que aporta énfasis sin recurrir a Primary).

Secondary

Accent color role · jerarquía media. Secondary es el Agile Blue de Embassy: el color de interacción y soporte, con menos énfasis que Primary. Se usa para los componentes menos prominentes de la UI —links, chips de filtro, acciones de apoyo, estados de navegación y elementos interactivos secundarios—. Donde Primary marca la acción principal de la pantalla, Secondary marca lo complementario y no debe competir como un segundo CTA dominante.

Ejemplos. Los links y los estados de navegación usan Secondary + On Secondary; los chips de filtro, los tags y los badges de apoyo usan Secondary Container + On Secondary Container (un fill tonal de menor énfasis). El badge "Borrador" es un buen ejemplo de Secondary Container.

El rol Secondary main se reserva para links, focus rings y estados activos.

Tertiary

Accent color role · jerarquía baja. Tertiary es el accent color violeta de Embassy y el de menor jerarquía. Es un color de contraste que se usa con intención para diferenciar y enfatizar áreas puntuales —una etiqueta "Nuevo", un tag destacado, una sección o un acento decorativo—. No compite con Primary ni Secondary; aparece para llamar la atención sobre algo específico. Si se usa de más, pierde su efecto.

Ejemplos. Los tags y destacados sólidos usan Tertiary + On Tertiary; los chips/badges tonales y los callouts de novedad usan Tertiary Container + On Tertiary Container.

Components · Actions

Button

Botones para acciones primarias, secundarias y de peligro.

Los botones permiten a las personas realizar acciones y tomar decisiones con un solo toque. En Embassy cada botón comunica el nivel de énfasis de su acción a través de un sistema de cinco variantes, construidas íntegramente con tokens semánticos --color-* para que funcionen sin ajustes en light y dark mode.

Cuándo usar botones

Usar para

  • Disparar una acción: guardar, crear, enviar, confirmar
  • Avanzar o retroceder en un flujo o wizard
  • Acciones destructivas con confirmación explícita
  • Acciones dentro de tarjetas, modales y toolbars

Evitar para

  • Navegar a otra página → usar un link
  • Acciones inline dentro de un párrafo → usar link text
  • Más de una acción primaria por vista
  • Alternar un estado on/off → usar switch o tabs

Máximo énfasis. La acción principal y más probable de una vista. Una sola por pantalla. Rol: --color-primary.

Secondary · Filled Tonal

Énfasis medio, claramente distinto del Primary. Botón tonal que consume Secondary Container + On Secondary Container (estilo MD3 filled-tonal): light periwinkle claro #CAD9FF + texto navy · dark periwinkle medio #3A5BB0 + texto claro. Se diferencia del Primary por inversión de valor. Tokens: --button-secondary-*.

Tertiary · Outlined

Énfasis medio. Acción alternativa o secundaria (cancelar, volver). Rol: --color-outline + --color-primary.

Text

Bajo énfasis. Acciones terciarias, enlaces de acción, áreas densas. Rol: --color-primary sin contenedor.

Icon

Acción compacta solo-ícono para toolbars y filas densas. Requiere aria-label.

Modificadores semánticos

Sobre Primary: .btn-danger para acciones destructivas y .btn-success para confirmaciones positivas.

Ejemplo interactivo

Composición real: jerarquía de acciones en el pie de un formulario. Una primaria, una terciaria de escape.

Jerarquía: una acción primaria por vista

Combiná variantes para guiar la atención. Nunca dos primarias compitiendo en la misma área.

✓ Correcto
Una primaria clara + una terciaria de escape. Jerarquía evidente.
✕ Incorrecto
Dos primarias compiten; el usuario no sabe cuál es la acción esperada.

Contenido del label

✓ Do
Verbos de acción concretos que describen el resultado.
✕ Don't
Labels genéricos o ambiguos no comunican qué pasará.
✏️Recomendaciones de contenido: verbo + objeto (“Crear vacante”), máximo 1–3 palabras, sin punto final, capitalización tipo oración, sin MAYÚSCULAS sostenidas.

Buenas prácticas

Hacer

  • Ordenar acciones: primaria a la derecha, escape a la izquierda
  • Usar Tertiary/Text para acciones secundarias, no otra Primary
  • Usar .btn-sm en tablas y filas densas
  • Reservar .btn-danger solo para acciones destructivas

Evitar

  • Más de una Primary visible por sección
  • Deshabilitar sin explicar por qué (usar tooltip)
  • Íconos decorativos que no aportan significado
  • Mezclar tamaños distintos en un mismo grupo

Errores comunes

Primary múltiple

Varias primarias en pantalla diluyen la jerarquía. Dejá una sola.

Danger por defecto

El rojo solo para acciones irreversibles, no para “cancelar”.

Icon sin label

Un icon-btn sin aria-label es invisible para lectores de pantalla.

Foco y teclado

Foco visible

Anillo de 2px (--color-focus) + halo (--color-focus-ring) vía :focus-visible. Visible solo en navegación por teclado.

Navegación por teclado

  • Tab — mueve el foco al siguiente botón
  • Shift + Tab — foco al anterior
  • Enter / Space — activa el botón enfocado
  • Orden de tabulación = orden visual del DOM

Lectores de pantalla y touch

  • ⌨Elemento nativo. Usar siempre <button> (no <div>) para obtener rol, foco y activación por teclado gratis.
  • 🔊Icon buttons. Requieren aria-label descriptivo: <button class="icon-btn" aria-label="Editar">. El label debe describir la acción, no el ícono.
  • 🚫Disabled. El estado es anunciado por el lector. Para acciones bloqueadas temporalmente, agregá un tooltip explicando por qué.
  • 👆Área táctil. Mínimo recomendado 44×44px (WCAG 2.5.5). El icon-btn de 36px debe llevar padding o margen extra en interfaces táctiles.

import * as React from "react"
import { cn } from "@/lib/utils"

export interface InputProps
  extends React.InputHTMLAttributes<HTMLInputElement> {
  label?: string
  required?: boolean
}

const Input = React.forwardRef<HTMLInputElement, InputProps>(
  ({ className, label, required, type, ...props }, ref) => {
    const input = (
      <input
        type={type}
        ref={ref}
        className={cn("field-input", className)}
        {...props}
      />
    )

    if (!label) return input

    return (
      <div className="field-group">
        <label className="field-label">
          {label}
          {required && <span className="field-required">*</span>}
        </label>
        {input}
      </div>
    )
  }
)
Input.displayName = "Input"

export { Input }

  open?: boolean
  onClose?: () => void
}

function Modal({ open, onClose, className, children, ...props }: ModalProps) {
  React.useEffect(() => {
    if (!open) return
    const handleEscape = (e: KeyboardEvent) => {
      if (e.key === "Escape") onClose?.()
    }
    document.addEventListener("keydown", handleEscape)
    return () => document.removeEventListener("keydown", handleEscape)
  }, [open, onClose])

  if (!open) return null

  return (
    <div className="modal-overlay" onClick={onClose}>
      <div
        className={cn("modal", className)}
        onClick={(e) => e.stopPropagation()}
        {...props}
      >
        {children}
      </div>
    </div>
  )
}

Guidelines · Illustrations

Illustrations

Las ilustraciones de Amalgama son un complemento visual para estados vacíos, onboarding y comunicaciones. Deben reforzar el tono humano, claro y profesional de la marca.

Estilo de ilustración

Línea limpia

Trazos de 2px con esquinas redondeadas. Sin sombras internas complejas. Consistente con el stroke de los íconos Lucide.

Paleta de marca

Colores del sistema: primary navy (#01164D), secondary blue (#4F80FF), tertiary purple (#7158F4). Nunca colores fuera de la paleta definida en Figma.

Personas inclusivas

Representación diversa. Evitar rasgos muy específicos. Figuras abstractas-humanas son preferibles para mayor inclusividad.

Fondos simples

Preferir fondos planos o degradados suaves. Las ilustraciones deben verse bien sobre white y sobre --neutral-50.

Styles · Motion

UI Animations

El motion en Amalgama es funcional, no decorativo. Cada animación sirve para comunicar estado, guiar la atención o confirmar una acción. Nunca para impresionar.

Principios de motion

⚡
Rápido y útil
Las animaciones de UI deben sentirse instantáneas. 120–200ms es el rango adecuado para la mayoría.
🎯

La animación debe guiar los ojos del usuario hacia lo relevante, no distraer de la tarea principal.
🔄
Consistencia
Elementos similares deben animarse de la misma manera. Usa los tokens de duración y easing del sistema.
♿

Focus management

Focus visible: Todos los elementos interactivos deben tener un focus ring visible. No elimines el outline con outline: none sin proporcionar un reemplazo. Usá :focus-visible para mostrarlo solo con teclado.

Focus trap en modales: Cuando un modal está abierto, el foco debe quedar atrapado dentro. Al cerrar, debe volver al elemento que lo abrió.

Design Tokens

Los design tokens son las decisiones de diseño de Embassy expresadas como variables con nombre. Son el puente entre el diseño y la implementación — una fuente de verdad compartida para diseñadores y desarrolladores.

¿Qué son los design tokens?

Un design token es un valor con nombre que representa una decisión de diseño: un color, un tamaño de tipografía, un radio de borde, una duración de animación. En lugar de hardcodear #01164D en cada componente, se usa --color-primary. Cuando cambia el color de marca, se actualiza el token una vez — y el cambio se propaga automáticamente a todos los componentes que lo usan.

Los design tokens conectan de manera significativa decisiones de estilo que, de otra forma, no tendrían una relación clara. Por ejemplo, si los mockups de un diseñador y la implementación de un desarrollador hacen referencia al mismo token para el "color de contenedor secundario", ambos pueden tener la seguridad de que se está usando el mismo color en los dos lugares. Esto aplica incluso si el valor hexadecimal asignado a ese token se actualiza.

✗ Valor hardcodeado 
.btn-primary  { background: #01164D; }
.badge-active { background: #01164D; }
.link         { color:      #01164D; }
.nav-item.on  { border:     #01164D; }
Si cambia el azul de marca, hay que buscar y reemplazar el hex en cada componente. Es fácil olvidarse alguno.
✓ Token 
.btn-primary  { background: var(--color-primary); }
.badge-active { background: var(--color-primary); }
.link         { color:      var(--color-primary); }
.nav-item.on  { border:     var(--color-primary); }
Se actualiza un solo token y el cambio se propaga a todos los componentes que lo usan.

¿Por qué son importantes los tokens?

Los tokens hacen posible que un design system tenga una única fuente de verdad: un repositorio donde las decisiones de estilo quedan registradas y los cambios pueden ser trackeados.

Como los tokens son reutilizables y están orientados a un propósito, pueden definir actualizaciones a nivel sistema para temas y contextos. Por ejemplo, se pueden usar tokens para aplicar sistemáticamente una paleta de alto contraste que mejore la visibilidad, o cambiar la escala tipográfica para asegurar que el texto sea legible en una pantalla de TV.

Al usar tokens tanto en diseño como en implementación, las actualizaciones de estilo se propagan de manera consistente en todo un producto o suite de productos. También ayudan a que diseñadores y desarrolladores "hablen el mismo idioma", reduciendo la confusión durante el handoff de diseño a implementación.

Consistencia
Todos los productos consumen la misma fuente de verdad. No hay interpretaciones diferentes del "azul de marca" entre equipos.
Escalabilidad
Un cambio en el token se aplica a todos los componentes. Dark mode, theming y rebrand sin tocar cada componente individualmente.
Handoff preciso
Diseñadores y devs hablan el mismo idioma. emb.color.primary en Figma es exactamente --color-primary en CSS.

Estructura de tokens en Embassy

Embassy usa dos niveles de tokens. Los tokens primitivos definen los valores crudos de la paleta; los tokens semánticos les asignan un propósito y son los que se aplican a los componentes.

Token primitivo
emb.color.primitive.navy-900
#01164D
El valor crudo de la paleta, sin contexto de uso. Define toda la paleta disponible para el sistema.
→
Token semántico
emb.color.primary
= emb.color.primitive.navy-900
Asigna un propósito al primitivo: el "para qué" (acción principal, fondo, texto). Es el token que se aplica a los componentes.
→
Componente
background: var(--color-primary)
El componente consume el token semántico. Nunca apunta a un primitivo ni a un valor hexadecimal directo.
Referencia externa: Material Design organiza sus tokens en tres niveles —.ref (reference), .sys (system) y .comp (component)—. Embassy se inspira en esa estructura pero la simplifica a dos niveles: primitivo y semántico. No usamos .ref, .sys ni .comp como parte de nuestro sistema.

Tipos de tokens

Embassy organiza sus tokens en seis categorías. Cada tipo tiene su propio conjunto de variables definidas en variables.css.

Color
Paletas primitivas y roles semánticos. --color-primary, --color-surface, --color-error. Ver sección Colors para la documentación completa.
Typography
Familias, tamaños, pesos y line-heights. --font-display (Epilogue), --font-body (Inter), --font-mono (DM Mono).
Spacing
Escala base-4. --space-1 (4px), --space-2 (8px), --space-4 (16px), --space-8 (32px), hasta --space-16 (64px).
Radius
Border-radius para cada nivel de componente. --radius-sm (4px), --radius-md (8px), --radius-lg (12px), --radius-xl (16px), --radius-full (9999px).

Contextos

Los tokens pueden apuntar a diferentes valores dependiendo de un conjunto de condiciones. Estas condiciones se llaman contextos y los valores resultantes se llaman valores contextuales. Ejemplos de contextos pueden incluir formatos de dispositivo, dark theme, layouts densos o sistemas de escritura de derecha a izquierda.

Se puede pensar un contexto como una especie de tag. Si el valor de un token está etiquetado como dark theme, entonces va a sobrescribir el valor default del token cuando el producto esté en un contexto de dark theme. Por ejemplo, emb.color.surface puede tener un valor para light mode y otro valor para dark mode, pero el componente sigue usando el mismo token.

Light mode
emb.color.surface
↓
#FFFFFF
Dark mode
emb.color.surface
↓
#13161F
El mismo token emb.color.surface resuelve a #FFFFFF en light mode y a #13161F en dark mode. El componente nunca cambia: sigue apuntando al token semántico, y el contexto decide el valor.

Usage guidelines

Do

  • Use Nav Cards for section-level navigation — overview grids, category landing pages, and shortcut anchors.
  • Always use a Lucide icon. Do not use emojis or custom illustrations in nav cards.
  • Keep title under 3 words and description under 8 words.
  • Use color variants purposefully — one or two colors per grid maximum to avoid visual noise.
  • Keep equal card heights across a grid (min-height handles this automatically).

Don't

  • Don't use Nav Cards as content cards (news, articles, profiles) — use .emb-card for editorial content.
  • Don't mix nav-card with the old emoji anchor pattern.
  • Don't override min-height per-card — inconsistent heights break grid alignment.
  • Don't use more than 6 color variants in a single grid.

Backward compatibility

All existing .ds-card-link / .ds-card-icon / .ds-card-name / .ds-card-desc HTML continues to work unchanged. The CSS for those classes has been updated to share the same token-driven visual design as .nav-card. No HTML migration is required for existing overview grids.
